# Cruise Captain

React Native (iOS + Android, single codebase) app for car club cruises (WVCA/NWCA). An organizer shares a route link; participants load it, get turn-by-turn guidance to each waypoint in sequence, and get auto-rerouted to the *next* waypoint (not the missed one) if they go off-route.

Full context: [docs/project-brief.md](docs/project-brief.md) (product scope) and [docs/architecture-spec.md](docs/architecture-spec.md) (technical shape). Read those before making scope or architecture decisions — this file is just the fast-recall summary.

## Key decisions already locked in

- **No backend for MVP.** Routes are static JSON files (organizer's own waypoint data, exported from Google My Maps) hosted at a URL and resolved via deep link. No accounts/login/payments/ads in MVP.
- **Online mode:** Google Maps Platform SDK for tiles + live tracking. "Turn-by-turn" is **our own lightweight guidance**, not Google's Navigation SDK: live GPS + straight-line distance/bearing to the current target waypoint, showing that waypoint's own `label` as the instruction (e.g. "Turn right on Hwy 18"). Decided this way because Google's actual Navigation SDK (with real road-routed turn-by-turn and auto-recalculation) has no mature React Native wrapper and would be a much heavier, riskier integration than this MVP calls for. This also means the same distance/bearing code doubles as the offline-mode fallback. See `src/routing/geo.ts` and `src/routing/useLiveNavigation.ts`.
- **The map's route *line*** is road-snapped via a one-time Directions API call per route load (`src/routing/directions.ts`), purely for display — falls back to straight lines between waypoints if that call fails. This is separate from and doesn't affect turn-by-turn guidance above, which always uses the raw waypoints. Uses the same Maps API key (`src/config/secrets.ts`, gitignored — copy from `secrets.example.ts`); turns out that key currently has no "Application restrictions" set (confirmed by calling the Directions API directly from a machine, not the app, and it worked), so it works for a plain REST call as-is. See the security follow-up below.
- **Offline mode:** MapLibre + OpenStreetMap raster tiles (`src/maps/OfflineMapProvider.tsx`), pre-fetched for the route corridor (zoom 11-15 — street-level only, country-level zooms are wasted tiles for a driving route) while still online via `OfflineManager.createPack` (`src/maps/offlineTilePrefetch.ts`). Guidance falls back to straight-line bearing/distance to the next waypoint — explicitly reduced-fidelity, not parity with online mode. (Google's ToS forbids caching Google tiles/routes offline — only the organizer's own waypoint data is cached.) Switches automatically via NetInfo connectivity state in `MapScreen.tsx`.
  - Gotcha: MapLibre's native HTTP layer can't parse `asset://` or `data:` URIs for `OfflineManager.createPack`'s style — needs a real `https://` URL. That's hosted at `hosting/osm-style.json` in this repo (why it now has a GitHub remote); the live `<Map>` component, by contrast, is fine with an inline style object (`osmStyleSpec` in `src/maps/osmStyle.ts`).
  - Gotcha: an `OfflinePack`'s `.metadata` is double-wrapped — the native layer nests whatever you pass under its own migration bookkeeping, so reading it back needs `pack.metadata.metadata` (itself still a JSON string), not `pack.metadata` directly. Got this wrong once already; it silently broke the "skip if already downloaded" dedup check and caused duplicate downloads.
- **The one custom piece of logic:** off-route reroute targets `waypoints[currentIndex + 1]` (skip ahead by `sequence`), never the missed waypoint. Implemented in `useLiveNavigation`: "off-route" is detected when the driver's live position becomes closer to the *next* waypoint than the current target by more than `OFF_ROUTE_SKIP_MARGIN_METERS` (30m) — a strong signal they drove past/missed the current turn. Same distance/bearing math powers both this and the plain arrival check, and both online and offline modes (once offline mode is built).
- **Monetization is undecided** but architecturally deferred behind a single `userHasAccess(routeId)` call at route-load time, which always returns `true` in MVP. Don't scatter access checks elsewhere.
- **Route data model:** see architecture-spec.md §2 — `sequence` (ordering), `type: start|waypoint|end` (drives UI, e.g. cruise-complete screen).
- Target users skew 50-70 — favor plain naming and low-friction UI over trendy/dense UI.

## Explicitly out of scope for MVP

In-app route creation/editing, accounts, payments, ads, multiple concurrent cruises, the My Maps→JSON converter tool (planning workflow is manual for now).

## Pre-launch follow-ups (not urgent for dev, do before shipping to real users)

- **Offline tiles come straight from OpenStreetMap's public tile server** (`tile.openstreetmap.org`), which explicitly isn't meant for production app traffic at scale. Fine for dev/testing; switch to a proper provider (MapTiler, Stadia Maps, etc., all have generous free tiers) before real users are on this — just update `hosting/osm-style.json`'s tile URL (and its API key if the provider needs one).
- **Google Maps API key has no "Application restrictions."** Confirmed by calling the Directions API directly from a non-Android machine with the key and it worked — meaning the key currently isn't locked to this app at all, it just works from anywhere. Any key embedded in a shipped app is eventually extractable from the APK regardless, so this isn't a new exposure, but it does mean a leaked key's blast radius is capped only by **"API restrictions"** on the key (in Cloud Console → Credentials), not by app identity. Before shipping: confirm "API restrictions" on the key is set to exactly the APIs in use (Maps SDK for Android, Maps SDK for iOS, Directions API) rather than "Don't restrict key" — that's the one thing actually protecting this key right now.

## Deferred enhancements (nice-to-haves, not MVP-blocking)

- **Granular turn-by-turn, not just pin-to-pin.** Found via a real driving test (2026-07-14, Duniway→Kreder→Fletcher→Monroe loop): the banner only announces the next organizer-placed pin, so any turns *between* pins (e.g. a turn onto Stoller Rd that wasn't its own pin) go unmentioned. Organizers are expected to keep routes to ~10 pins, but a real cruise can have 15-30 actual turns — the pin count and the turn count are different things. Fix doesn't need more pins: the Directions API call already made for road-snapping (`src/routing/directions.ts`) returns real turn-by-turn `steps[]` (e.g. "Turn right onto Stoller Rd") that are currently discarded, keeping only the polyline. Plan: drive the online banner off the nearest upcoming *step*, not the nearest pin; keep pins as-is for off-route skip-ahead logic and for the offline fallback (no network = no Directions API = today's simpler pin-to-pin behavior, already reduced-fidelity by design). Not yet built — real scope, needs its own pass.
- Tap the turn banner to zoom out and see the whole route + all pins/distances (like Waze's Android Auto view), tap again to collapse back to normal turn-by-turn. Came up during real-device testing on a Pixel 9 Pro XL — worth doing, just not now.
- **Android Auto support.** Confirmed not implemented — and it's not a small add. Android Auto for navigation apps requires Google's separate Car App Library (its own template-based rendering surface for driving safety, not a mirrored phone screen), which is distinct native Android work, not something react-native-maps or this app gets for free. Real project on its own if wanted.
- Onboarding/tutorial content, needed for two different audiences: (1) participants — quick "how this works" (tap link, follow banner), (2) organizers — how to plan a route in Google My Maps so it converts cleanly (name each pin with its turn instruction, use the "Directions" layer for the connecting line — see the real My Maps route tested 2026-07-14 for a worked example of what converts well). Needs a way to reach this from within the app (a help icon somewhere).
- Route "get to the start" leg: right now the app assumes the driver is already standing at the route's `start` pin at launch (guidance begins targeting the *second* waypoint). It doesn't currently help someone get from their actual current location (e.g. their house) to the route's official starting point.
- QR code as a second distribution channel alongside text/email links — same underlying deep link, just rendered as a scannable image for the organizer to share/print. No new app logic needed since it's the same URL; would need something (in-app or not) to generate the QR image from a route link.
- A My Maps → route-JSON converter (upload a My Maps share link, get a route link back). Still explicitly out of scope per the architecture spec, but worth prioritizing — a real My Maps route tested 2026-07-14 (start/end loop + 3 named turn pins, using My Maps' own "Directions" layer for the line) converted to our schema almost field-for-field with no rework needed, which is a strong signal this conversion is straightforward to automate.

## Open questions (not yet decided — flag before assuming)

- Where route JSON gets hosted for deep links to resolve against (GitHub raw file vs. a cloud storage bucket). Currently sidestepped: the deep link itself carries the route JSON's URL (`cruisecaptain://route?url=...`), so hosting can be decided later without an app change.
- Universal link (`https://...`) vs. custom scheme (`cruisecaptain://`) for the route link — universal + app-store fallback is the current leaning given the older target audience, but not decided. Custom scheme is what's implemented now (simplest, no hosting/domain dependency); switching to a universal link later is additive, not a rewrite.
- Arrival-radius threshold for "reached a waypoint" — defaulted to 50m (`ARRIVAL_RADIUS_METERS` in `src/routing/useLiveNavigation.ts`), not validated against real driving yet. Confirmed the intended behavior for pins the driver doesn't stop at (e.g. a rest stop set back from the road): the off-route skip-ahead logic already handles this — once the driver is closer to the *next* pin than the one they drove past, it advances forward on its own, no separate "drove by without stopping" case needed. Still worth revisiting the 50m number itself after more real driving — GPS update interval was tightened from 3s to 1s (2026-07-14, ahead of a real driving test) since 3s risked skipping past the arrival radius entirely at highway speed.
